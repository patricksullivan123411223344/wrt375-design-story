Can LLM NPCs Feel Real?

A human-centered design case study exploring memory, emotion, and believable AI-driven character interaction.

About the Student

Patrick Sullivan is a student and developer focused on building stateful AI systems that maintain memory, identity, and adaptive behavior across interactions.

This portfolio showcases a design case study for WRT 375: Human-Centered Design. The project explores how AI-driven NPCs can create more immersive, responsive, and human-like experiences in video games.

Project Abstract

This project explores how AI-driven non-player characters can create more immersive and human-like interactions in video games. Traditional NPCs often feel static, predictable, and emotionally disconnected because they do not remember past interactions or adapt meaningfully to player behavior.

To investigate this problem, I built a prototype NPC conversation system that reacts to user tone, tracks emotional state changes, and attempts to recall previous moments in the interaction. Participants tested the prototype through structured scenarios including first contact, hostile interaction, reconciliation, and memory recall.

User testing showed that players are not simply looking for the most advanced AI model. They are looking for an NPC that feels believable. Across both technical and non-technical testers, the main immersion breaks were response latency, hallucinated or drifting responses, and dialogue that felt too long or overly dramatic.

These findings led to a design pivot: instead of using the LLM to generate every line of dialogue in real time, the stronger approach is to use AI as a control layer that selects from authored dialogue, scripted events, and memory-aware responses. This keeps the experience adaptive while improving pacing, reliability, and immersion.

Design Problem

In most video games, NPCs feel static and predictable. They often repeat the same dialogue, forget previous interactions, and fail to respond emotionally to the player’s behavior.

This project was guided by the question: how might we design an NPC that remembers past interactions and responds dynamically based on emotional context in order to create a more immersive and human-like experience?

User Research

User testing was conducted across two groups: casual or non-technical users and technically inclined users from Computer Science backgrounds. This structure allowed the project to gather both baseline usability feedback and deeper technical feedback about realism, system limitations, and interaction quality.

Participants interacted with the NPC through structured scenarios: first contact, hostile interaction, reconciliation, and memory recall. These scenarios were designed to test whether users noticed emotional changes, memory behavior, and adaptive responses.

Design Process

The project began by identifying a common weakness in traditional NPC design: characters often fail to remember or meaningfully react to the player. From there, I developed a prototype focused on emotional responsiveness, memory tracking, and conversational adaptation.

Early iterations focused on building a working conversation loop where the NPC could respond to user tone and preserve context across turns. After testing, the design evolved based on feedback about latency, hallucinations, prompt drift, and unrealistic response length.

This process shifted the project from asking “Can an LLM generate dynamic NPC dialogue?” to a more useful design question: “How can AI support believable interaction without breaking pacing, trust, or immersion?”

System Overview

The prototype was designed as a stateful NPC system rather than a simple chatbot. Instead of treating each message as an isolated prompt, the system carries structured information forward across the interaction so the NPC can respond with a consistent personality, emotional state, and memory of prior events.

At the center of the system are structured context payloads that organize important information before it is sent into the model. These payloads help separate stable information, such as the NPC’s identity and personality, from changing information, such as the user’s tone, recent actions, emotional shifts, and conversation history.

The system also uses supporting instruction files to guide the NPC’s tone, response style, and role within the interaction. Short-term and longer-term memory patterns allow the NPC to behave more consistently across the conversation.

This architecture allowed the NPC to behave more consistently than a standard one-off LLM response. The goal was not just to generate dialogue, but to simulate a character that could remember, react, and change based on the user’s behavior.

What Users Found

Across both groups, the strongest finding was that users were not looking for the most advanced AI model. They were looking for an interaction that felt believable. The NPC needed to respond with a realistic cadence, maintain consistent character behavior, and avoid breaking immersion.

Latency Broke Immersion

When the NPC took too long to respond, users became more aware of the system behind the character. Even a delay of a few seconds made the interaction feel less natural.

Prompt Drift Broke Trust

Smaller local models sometimes drifted away from the intended personality or produced hallucinated responses, making the NPC feel inconsistent.

Long Responses Felt Unrealistic

Users responded better when the NPC had a clear conversational cadence. Overly long or dramatic responses made the character feel less like a real person and more like a chatbot.

These results showed that the main challenge was not simply making the NPC “smarter.” The real challenge was designing a system that could preserve pacing, consistency, and emotional believability during interaction.

The Design Pivot

The testing results showed that the original design gave the language model too much control over the moment-to-moment interaction. While live generation made the NPC feel flexible, it also introduced problems with latency, hallucination, prompt drift, and overly long responses.

Because of this, the next version of the system shifts from a generation-first model to a selection-first model. Instead of asking the LLM to write every line of dialogue in real time, the system would use authored dialogue, scripted events, and memory-aware response options.

Original Approach

The LLM generates the NPC’s dialogue directly during the interaction.

This creates flexibility, but also increases response time and makes the character less reliable.

Improved Approach

The LLM acts as a control layer that selects from prepared dialogue and scripted behavior.

This keeps the NPC adaptive while improving pacing, consistency, and immersion.

This pivot reframes the role of AI in NPC design. The goal is not to replace authored game writing, but to enhance it with memory, emotional state, and contextual decision-making.

Outcomes & Takeaways

This project demonstrated that increasing AI complexity does not automatically improve user experience. In real-time interaction systems, pacing, consistency, and believability matter more than raw model capability.

The most important takeaway is that AI should support interaction design, not replace it. By shifting from generation to selection, the system becomes more reliable while still maintaining adaptability.

In future iterations, I would explore more advanced memory systems, real-time optimization techniques, and a larger library of authored dialogue paths to further improve responsiveness and immersion.
